## 1. 摘要與目標

NFC 門禁刷卡（兩個辦公室門口）作為**打卡時間的備援來源**。若員工忘記（或無法）在手機／
WebAuthn PWA 上打卡，但實際上有在門口刷卡，該筆刷卡紀錄會**補上缺漏的打卡**。若員工在手機
上已經打過卡，則以**手機紀錄為準** —— NFC 絕不與之競爭。

補登為**單邊（per-side）**：只補上缺少的那一邊（上班或下班）。這是**補登（gap-fill）**，
不是合併（merge）。

系統現有的所有功能（WebAuthn PWA 打卡、定位、每月補登、報表、台灣行事曆等）**維持不變**。
本功能只新增一條獨立、隔離的備援匯入路徑。

| 網路 | 門禁主機位於**辦公室 LAN**（私有 IP、在 NAT 後方）。正式機（`go2fresh-1`）相對於該 LAN 為**遠端**。 |

**部署備註（2026-07-01 已確認）：**
- 門禁主機的 IP 為 **DHCP 動態配發、可能變動** —— 這正是推送代理（push-agent）設計「不依賴

  位於 `192.168.2.0/24`，與開發設定白名單 `allowedDevOrigins: 192.168.2.*` 為同一網段，因此

- `card_serial`（第 5 欄）為實體 NFC 卡片被讀取到的**卡片 UID**（僅供參考，匯入邏輯不使用）。

## 3. `YYYYMM.txt` 格式規格

以逗號分隔，一列一筆刷卡，無標題列：

```
date, time, emp_id, door_no, card_serial, name
```

| # | 欄位 | 範例 | 格式 | 使用？ |
|---|------|------|------|--------|
| 1 | date | `20260701` | `YYYYMMDD` | ✅ |
| 2 | time | `072437` | `HHMMSS`（24 小時制） | ✅ |
| 3 | emp_id | `F1000118` | 對應 `employees.emp_id` | ✅（比對鍵） |
| 4 | door_no | `1` / `2` | 辦公室／門號 | ℹ️ 僅記錄 |
| 5 | card_serial | `5717003342` | 實體**卡片 UID**（SOYAL 讀取） | ℹ️ 僅記錄 |
| 6 | name | `王小明` | 員工姓名，**CP950** | ℹ️ 僅記錄 |

原始資料範例（若以 UTF-8 讀取，姓名會變成亂碼 —— 這正是檔案為 CP950、必須以
`encoding="cp950"` 解碼的判斷依據）：

```
20260701,072437,F1000118,1,5717003342,<CP950 姓名位元組>
20260701,074042,F1000818,1,5716344222,<CP950 姓名位元組>
20260701,081045,F1000220,2,6108331019,<CP950 姓名位元組>
```

**重要：** `door_no` **不是**上下班方向，而是標示刷卡發生在兩個辦公室中的哪一個。因此上／下班
是以**時間**推斷，而非此欄位（見 §5）。

## 4. 整合架構 —— 兩邊如何連接

### 限制條件
門禁主機為 **NAT 後方的私有 LAN IP**，正式機（`go2fresh-1`）為遠端。網際網路 →
私有 LAN 的**入站**連線會被阻擋；**LAN → 網際網路的出站**連線則被允許。因此「正式機主動去拉
門禁主機的檔案」是困難／被阻擋的方向。

### 選定方案 —— **推送代理（push agent）**
由 **LAN 端主動推送**檔案到正式機既有的公開 HTTPS 端點，**不需**任何 VPN、通訊埠轉發或防火牆
變更。

```
門禁主機（辦公室 LAN）                          正式機（go2fresh-1, Docker）
────────────────────                          ────────────────────────────
00:10  門禁系統寫入 YYYYMM.txt
00:20  工作排程執行 push-nfc.ps1：              POST /api/nfc/import
         以原始位元組讀取 YYYYMM.txt  ────────►（X-NFC-API-Key, HTTPS）
         POST 至正式機 API                        │ 解碼 cp950
                                                   │ 解析各列
                                                   │ 單邊補登（具冪等性）
                                                   │ 重新產生每日出勤摘要
                                                   └─► JSON 報告 ──► 由腳本記錄
```

所有解析／解碼／補登／摘要邏輯都放在**後端**（納入版本控管、可測試）。門禁端腳本刻意保持簡單：
讀檔 → POST。

**v1 已評估但不採用的替代方案：** OneDrive／Microsoft Graph 雲端拉取（需 Azure 應用程式與
OAuth 維護）；VPN／通道（如 Tailscale，為了一個每天一次的小檔案得長期維運基礎設施）。兩者仍是
未來可行的乾淨選項。

## 5. 上／下班推斷（以時間為準）

針對每個 `(emp_id, date)`，就當天所有刷卡（含兩道門）：
- **最早的刷卡 → 上班打卡候選**
- **最晚的刷卡 → 下班打卡候選**

邏輯上忽略 `door_no`。員工可能早上在 1 號辦公室刷卡、晚上在 2 號辦公室刷卡 —— 以時間仍能正確
判定上／下班。此做法與系統既有的**最早進、最晚出（First-In-Last-Out）**模型一致。

---

## 6. 單邊補登演算法（核心）

針對檔案中每個 `(emp_id, date)`：

| 當天既有的**真實**打卡 | 動作 |
|---|---|
| 0 筆（整天皆空） | 補上 NFC 上班（最早刷卡）。若刷卡 ≥ 2 筆，另補 NFC 下班（最晚刷卡）。 |
| 1 筆（有上班、缺下班） | **僅**補上 NFC 下班（最晚刷卡），*且需晚於既有的上班時間*。 |
| 2 筆以上（完整） | 不動作 —— 以手機為準。 |

規則：
- **「既有打卡」** = 當天未被覆寫（non-overridden）的 `attendance_logs`（手機打卡**與**先前
  由 NFC 補登的紀錄都算）。
- 每次**最多補一筆上班、一筆下班**，且只補缺少的那一邊，因此 NFC **永遠不會取代**真實的手機
  打卡（真正的補登，而非合併）。
- **下班時間護欄：** 下班候選只有在*嚴格晚於*有效上班時間時才採用。可避免「下班早於上班」，也
  防止單一刷卡被硬湊成一筆下班。
- **最短工時護欄：** v1 **關閉**（已決定）。（未來可調：僅當 `最晚 − 最早 ≥ N` 分鐘時才補下班，
  以降低誤判 EARLY_LEAVE。）

**已知限制（可接受）：** 僅憑門禁時間戳，若某人早上刷了兩次、離開時卻沒刷卡，最晚的那筆上午刷卡
會被當成當天「下班」（顯示為 EARLY_LEAVE）。這是門禁資料本身的先天限制。員工可透過既有的
**每月補登（Monthly Override）**自行修正 —— 與現況相同的安全閥。

---

## 7. 冪等性（可安全重複匯入）

當月檔案每天以累加內容重送。由於「既有打卡」已包含先前由 NFC 補登的紀錄，已補過的那一邊在重跑時
會直接略過。此外，寫入時另以 `emp_id + 完全相同的時間戳` 去重。**匯入 N 次的結果 = 匯入 1 次的
結果。** 人工／HR 覆寫永遠優先（覆寫會將先前紀錄標記為 `is_overridden=True`，NFC 隨後即以覆寫
後的紀錄為既有打卡而略過）。

## 8. 後端設計

遵循專案分層模式（router → service → repository；router 不直接存取 DB；全面 async；型別註記；
不可變結果型別）。

- **Router** `backend/app/routers/nfc.py`
  - `POST /api/nfc/import` —— 接收原始檔案內容（CP950 位元組）。以 API 金鑰驗證（見 §9），
    回傳匯入報告（見 §10）。
- **Service** `backend/app/services/nfc_import_service.py`
  - 解碼 cp950 → 解析各列 → 依 `(emp_id, date)` 分組 → 套用 §6 → 以行事曆準確的 `day_kind`
    重新產生每日摘要（重用 `reporting_service.generate_daily_summary`）→ 組出報告。
- **Repository** —— 重用 `attendance_repository`（`create_log`、`find_by_employee_and_date`）
  與 `summary_repository`。僅在需要 NFC 精確時間戳去重時，才新增小型輔助函式。
- **寫入紀錄的來源標記：** NFC 補登的 log 使用 `ip_address="nfc"`、`work_mode=OFFICE`、
  `latitude=longitude=accuracy=0`、`is_overridden=False`。重用既有慣例（大量補登用
  `ip_address="override"`）。*（替代方案 —— 正式的 `source` 欄位 —— 為避免對核心資料表做
  migration 而延後；日後要升級為正式欄位很容易。）*

---

## 9. 安全性

- 端點以 **API 金鑰**驗證，置於 HTTP 標頭 `X-NFC-API-Key`，與環境變數
  **`NFC_IMPORT_API_KEY`** 比對（加入 `backend/.env.example` 與
  `backend/.env.production.example`；**絕不寫死**；真實值僅存在於正式機 `.env` 與門禁主機的
  排程工作中）。
- 僅走 HTTPS。重用既有的速率限制。此為機器對機器（machine-to-machine），**不用** JWT
  （門禁腳本並非登入中的使用者）。
- 金鑰無效／缺漏 → `401`。未知的 `emp_id` 予以略過、不致命（無使用者列舉疑慮 —— 此為內部可信
  資料饋送）。

## 10. 報告 / 可觀測性

`POST /api/nfc/import` 回傳 JSON：

```json
{
  "filled_in": 0,
  "filled_out": 0,
  "skipped_already_punched": 0,
  "skipped_already_imported": 0,
  "unknown_emp_ids": [],
  "parse_errors": [],
  "affected_days": []
}
```

門禁腳本會逐行將其記錄於本機。*（持久化的 `nfc_import_runs` 稽核資料表列為 v2 加值項目。）*

## 11. 門禁端推送腳本

- **`tools/nfc-agent/push-nfc.ps1`**（PowerShell —— Windows 免安裝）搭配一份簡短 **README**，
  內容涵蓋：註冊工作排程、API 金鑰放置位置、正式機端點 URL。
- 行為：計算當月檔名 → 以**原始位元組**讀取 `YYYYMM.txt`（不在客戶端解碼）→ 帶
  `X-NFC-API-Key` POST 至正式機。**每月 1 號**另外重送**上個月**的檔案（以補齊上月最後一天的
  晚間刷卡）。網路失敗時重試；於本機附加一行紀錄。
- 排程：**每天約 00:20**（於 00:10 產生檔案之後）。

---

## 12. 假日 / 週末處理

檔案中**每一天**的刷卡都會被記錄（已決定：記錄所有刷卡，含週日）。摘要重算採用台灣行事曆的
`day_kind`，因此週末／假日的工作會被判為 **NORMAL**（絕不會是 LATE/EARLY_LEAVE），與既有規則
（#31f）一致。此處刻意**不**複製面向人員操作的週日「例假日」寫入鎖 —— NFC 記錄的是客觀的硬體
事實證據。

---

## 13. 邊界情況

- **未知 emp_id**（檔案中有、`employees` 中無）→ 略過並加入 `unknown_emp_ids`。絕不使整批
  失敗。
- **離職員工** → 略過補登，於報告中註記。
- **格式錯誤／過短的列／日期時間無效** → 略過並收集至 `parse_errors`。
- **跨月** → 由每月 1 號的雙檔重送處理（見 §11）。
- **空檔或檔案不存在** → 腳本記錄並以 0 結束（no-op）；端點對空 body 亦優雅處理。

## 14. 測試（TDD，目標 80%+）

**後端單元測試**（`nfc_import_service`）：
- CP950 解碼（Big5 fixture 的姓名可正確還原）。
- 各列解析含格式錯誤 → `parse_errors`。
- §6 補登：整天皆空（2 筆刷卡 → 上＋下；1 筆刷卡 → 僅上班）、僅有上班的日子（補下班）、
  完整日子（no-op）。
- 下班時間護欄（最晚 ≤ 上班 → 不補下班）。
- 冪等重複匯入（第二次執行不寫入任何資料）。
- 未知 emp_id 與離職 emp_id 的處理。
- 摘要重算產生正確狀態，含週末 = NORMAL。

**整合測試**（`POST /api/nfc/import`）：
- 有效金鑰 + 範例 CP950 檔案 → 預期報告 + 產生的紀錄。
- 缺漏／無效金鑰 → `401`。

## 17. 待議 / v2 構想

- 持久化匯入紀錄稽核表 + 小型 HR 檢視畫面。
- 將 `ip_address="nfc"` 升級為一級的 `source` enum 欄位。
- 若誤判 EARLY_LEAVE 過多，啟用可選的最短工時護欄（見 §6）。
- 若需當天即時可見，增加每天第二次推送（例如中午）。
